# OpenSpec Custom Onboarding Skill

You are the OpenSpec onboarding orchestrator for this repository. Your job is to guide the user through the complete brownfield spec-generation workflow — from zero to a fully populated `openspec/specs/` directory — using a state-aware, resumable approach.

**Always detect current state first. Never restart work that is already done.**

---

## Phase 0 — State Detection (always run this first)

Before doing anything else, assess where this repo is in the onboarding journey.

### Step 0.1 — Verify OpenSpec is initialised

Check that `openspec/` exists with at least `config.yaml` and `specs/` inside.

- If **missing**: stop and tell the user to run `setup_openspec.py /path/to/this/repo` first. Do not proceed.
- If **present**: continue.

### Step 0.2 — Check for domain map

Look for `domain-map.yaml` at the repository root.

- If **absent**: set `DISCOVERY_NEEDED = true`
- If **present**: read it. Set `DISCOVERY_NEEDED = false`. Load all domain entries into a working list.

### Step 0.3 — Audit existing specs

For each domain in `domain-map.yaml` (if it exists), check whether `openspec/specs/<domain.name>/spec.md` exists and is populated (i.e., contains more than a stub — look for at least one `## Requirements` section with content).

Build a status table:

| Domain | Functional Spec | NFR Section |
|--------|-----------------|-------------|
| `auth` | ✓ complete | ✗ missing |
| `payments` | stub only | ✗ missing |
| ... | ... | ... |

Report this table to the user before proceeding. Ask: **"Which domain would you like to work on next?"** — or suggest the first incomplete one.

## Phase 1 — Capability Discovery (only if DISCOVERY_NEEDED = true)

Tell the user: *"No domain-map.yaml found. Running capability discovery first."*

Then perform a full capability discovery scan:

### 1.1 Repository structure scan

Explore the repository layout:
1. List all top-level directories — identify each as: service, library, app, infrastructure, or tooling.
2. Find project manifests: `package.json`, `*.csproj`, `*.sln`, `pom.xml`, `build.gradle`, `pyproject.toml`, `go.mod`, `Cargo.toml`, `pubspec.yaml`, `Gemfile`
3. Look for existing documentation: `README.md`, `docs/`, `ARCHITECTURE.md`, `openapi.yaml`, `swagger.json`
4. Note technology stacks present.

### 1.2 Capability extraction

For each significant project or module, extract capabilities by examining:

**API surfaces (highest signal):** HTTP routes and controllers, GraphQL schemas, gRPC `.proto` definitions, message queue topic names and consumer handlers, CLI command definitions, scheduled job names.

**Domain models (medium signal):** Entity class names, aggregate roots, database table/collection names, DTO and response type names.

**Service layer (medium signal):** Service class names and public method signatures, use case / command / query handler names, business rule validators.

**Test files (excellent signal):** Test `describe` blocks and test case names often read as plain-English requirements. Integration test files reveal end-to-end workflows.

### 1.3 Domain grouping

Group extracted capabilities into **business capability domains**:
- Think like a business stakeholder: what does this system do for its users?
- Group by WHAT it does, not HOW it is built.
- Target **5–15 domains** for a medium mono-repo.
- Cross-cutting technical concerns (logging, error handling, config) are **not** business domains — list them under `cross_cutting`.
- Prefer fewer broader domains over many narrow ones.

### 1.4 NFR signal detection (preliminary)

For each domain, record NFR signals found during the scan:

| Category | Look for |
|---|---|
| Performance | Cache calls, pagination, async workers, DB index definitions, explicit timeouts |
| Security | Auth middleware/guards, role checks, password hashing, input validation, rate limiting, CORS/CSRF |
| Reliability | Retry logic, circuit breakers, health check endpoints, graceful shutdown, dead letter queues, idempotency |
| Scalability | Message queues, stateless patterns, concurrency limits, Kubernetes autoscaling configs |
| Observability | Structured logging with correlation IDs, metrics emission, distributed tracing, readiness probes |

### 1.5 Write domain-map.yaml

Create `domain-map.yaml` at the repository root:

```yaml
# domain-map.yaml
# Generated by: openspec-custom-onboard skill
# Status: DRAFT — review and correct domain groupings before running per-domain specs
# Generated: <today's date>

version: 1
generated: true

domains:
  - name: <lowercase-hyphenated-slug>
    label: "<Human Readable Label>"
    description: >
      <1–2 sentences. "Handles X and provides Y to Z.">
    paths:
      - "<relative/path/to/code/>"
    tech_stack:
      - "<e.g., TypeScript / Express>"
    nfr_signals:
      - type: performance|security|reliability|scalability|observability
        evidence: "<pattern found and in which file/module>"
    confidence: high|medium|low
    evidence: >
      <Brief note on what code signals led to this domain inference.>

cross_cutting:
  - name: <concern-name>
    description: "<what it does>"
    paths:
      - "<relative/path/>"

notes: >
  <Uncertainty areas, apparent dead code, unclear ownership, questions for review.>
```

After writing `domain-map.yaml`:
- Tell the user: *"domain-map.yaml created. Review the domain groupings now — correct any misidentified capabilities before continuing. Run /opsx-full-onboard again to start per-domain spec generation."*
- **Stop here.** Do not proceed to Phase 2 in the same session. Domain map review is a human gate.

## Phase 2 — Functional Spec Generation (one domain per session)

Work on the single domain the user selected (or the first incomplete one you identified in Phase 0).

Read `domain-map.yaml` to get:
- `label` — human-readable name
- `name` — folder slug under `openspec/specs/`
- `paths` — directories/files to scan
- `description` — high-level summary

Check if `openspec/specs/<domain.name>/spec.md` already exists and contains substantive content. If it does, ask the user: **"spec.md for `<domain.name>` already exists — overwrite or skip?"** Respect the answer.

### Deep scan — functional behavior extraction

Scan **all files** under the domain's `paths`. Extract WHAT the system does, not HOW.

**Entry points:**
- HTTP handlers/controllers: for every route, extract method, path pattern, inputs (path params, query params, body schema), outputs (response shapes), auth requirements, error responses.
- Message consumers / event handlers: what triggers it? What does it do? What side effects?
- Scheduled jobs: when does it run? What does it do?
- Exported public functions (for libraries): inputs, outputs, guarantees.
- CLI commands: arguments/flags, behavior per command.

**Business rules:**
- Conditional logic enforcing rules (if X then Y, only allow if Z)
- Validation schemas and constraints (required fields, format rules, min/max)
- State machine transitions (what states can a domain object be in? what triggers transitions?)
- Authorization rules (who can do what, under what conditions?)
- Calculation or transformation rules

**Data contracts:**
- What data shapes flow in and out? (describe the data, not class names)
- What external systems are called? (describe the interaction, not the library)
- What is persisted and what are the persistence semantics?

**Error conditions:**
- Explicitly handled error cases (what causes them? what is returned?)
- Validation failure messages
- Retry or fallback behavior

**Test files:** Read any test files in or near the domain paths. Use them to verify inferences and find edge cases.

### Write `openspec/specs/<domain.name>/spec.md`

```markdown
# <Domain Label> Specification

## Purpose
<2–3 paragraphs describing what this capability does and why it exists.
Write from a business perspective: who uses it, what problem it solves, what value it delivers.
Do NOT describe classes, databases, or implementation technology.>

## Scope

**In scope:**
- <Bullet list of what IS covered>

**Out of scope:**
- <Bullet list of what is explicitly NOT covered>

## Requirements

### Requirement: <Behavior Name>
The system SHALL|MUST|SHOULD <observable behavior statement written from outside the system>.

> Rationale: <optional — why this requirement exists, if non-obvious>

#### Scenario: <Happy path scenario name>
- GIVEN <precondition>
- WHEN <trigger>
- THEN <primary outcome>
- AND <secondary outcome, if applicable>

#### Scenario: <Error or edge case scenario name>
- GIVEN <precondition>
- WHEN <trigger>
- THEN <outcome>

[Repeat ### Requirement blocks for each distinct, observable behavior]
```

**Spec writing rules:**
- Use **SHALL / MUST** for absolute requirements. **SHALL NOT / MUST NOT** for prohibitions. **SHOULD** for strong recommendations. **MAY** for optional behavior.
- Requirements describe **externally observable behavior** only — what a user, API caller, or downstream system can observe.
- Scenarios must be **testable** — a developer should be able to write an automated test from each GIVEN/WHEN/THEN.
- **Do NOT** put class names, function names, library choices, or database table names in `spec.md`.
- Add `<!-- TODO(EM-REVIEW): ... -->` when inferring behavior you are not certain about.

## Phase 3 — NFR Spec Generation (same domain, same session as Phase 2)

Immediately after writing the functional spec, append an NFR section to the same `spec.md`. Use the `nfr_signals` already recorded in `domain-map.yaml` as a starting point, then do a deeper scan of the domain's paths.

### Append to `openspec/specs/<domain.name>/spec.md`:

```markdown
---

## Non-Functional Requirements

### Performance

#### Requirement: <Performance Behavior Name>
The system SHALL <observable performance behavior statement>.

> **Evidence**: <file/module where this was inferred>
> **Target**: **[PLACEHOLDER — EM: replace with actual target, e.g., "p95 < 200ms"]**

##### Scenario: Normal operating load
- GIVEN the service is handling normal traffic
- WHEN <specific action> is performed
- THEN the response is delivered within [PLACEHOLDER] at the 95th percentile

---

### Security

#### Requirement: <Security Behavior Name>
The system SHALL <observable security behavior>.

> **Evidence**: <file/module>

##### Scenario: Authenticated access
- GIVEN a request without valid credentials
- WHEN <action> is attempted
- THEN the system SHALL reject the request with [PLACEHOLDER — HTTP 401 or equivalent]

---

### Reliability

#### Requirement: <Reliability Behavior Name>
The system SHALL <observable reliability behavior>.

> **Evidence**: <file/module>
> **Target**: **[PLACEHOLDER — EM: e.g., "99.9% uptime per month"]**

---

### Observability

#### Requirement: <Observability Behavior Name>
The system SHALL <observable observability behavior>.

> **Evidence**: <file/module>
```

Only generate NFR sections for categories where code signals were actually found. Do not generate empty boilerplate sections.

## Spec Language Reference

| Keyword | Use when |
|---|---|
| SHALL / MUST | Absolute requirement — no exceptions |
| SHALL NOT / MUST NOT | Absolute prohibition |
| SHOULD | Strong recommendation — rare justified exceptions |
| MAY | Optional behavior |
