Run the **full OpenSpec brownfield onboarding workflow** for this repository. This command is state-aware — it detects what has already been done and continues from where you left off.

## Step 1 — State Detection

**Do this before anything else. Never skip.**

### 1a. Verify OpenSpec is initialised

Check that `openspec/` directory exists at the repo root with `config.yaml` inside.

- If **missing**: stop immediately. Tell the user:
  > "OpenSpec has not been initialised in this repository. Run `setup_openspec.py /path/to/this/repo` first, then re-run `/opsx-full-onboard`."
  Do not proceed.

### 1b. Check for domain-map.yaml

Look for `domain-map.yaml` at the repository root.

- **Not found** → set `NEEDS_DISCOVERY = true`
- **Found** → read it, set `NEEDS_DISCOVERY = false`, load all domain entries

### 1c. Audit existing specs

For each domain in `domain-map.yaml` (if it exists), check whether `openspec/specs/<domain.name>/spec.md` exists and contains substantive content (at least one `## Requirements` section with at least one requirement written, not just a stub header).

Also check whether the spec contains a `## Non-Functional Requirements` section.

Report a status table to the user:

```
Domain onboarding status:
  ✓  auth          — functional spec ✓, NFR ✓
  ○  payments      — functional spec ○, NFR ○
  ◑  notifications — functional spec ✓, NFR ○
  ...
```

Then ask:
> "Which domain would you like to work on? (Suggested next: `<first incomplete domain>`)"

Wait for the user's answer before proceeding. If the user says "suggested" or similar, proceed with the first incomplete domain.

## Step 2 — Capability Discovery (only if NEEDS_DISCOVERY = true)

Tell the user: *"No domain-map.yaml found — running capability discovery first. This is a one-time step."*

### 2.1 Repository structure scan

1. List all top-level directories; classify each as: service / library / app / infrastructure / tooling.
2. Find project manifests: `package.json`, `*.csproj`, `*.sln`, `pom.xml`, `build.gradle`, `pyproject.toml`, `go.mod`, `Cargo.toml`, `pubspec.yaml`, `Gemfile`.
3. Look for existing documentation: `README.md`, `docs/`, `ARCHITECTURE.md`, `openapi.yaml`, `swagger.json`.
4. Note technology stacks present across the repo.

### 2.2 Capability extraction

For each significant project or module, extract capabilities from:

**API surfaces (highest signal):**
- HTTP routes and controllers — the URL path reveals the domain (e.g., `/api/payments/` → payments domain)
- GraphQL schemas and resolver names
- gRPC `.proto` service and method definitions
- Message queue topic/channel names and their consumer handlers
- CLI command definitions and their descriptions
- Scheduled job names and handler function names

**Domain models (medium signal):**
- Entity class names, aggregate roots, value objects
- Database table/collection names (migration files, ORM models, schemas)
- DTO, request, and response type names and their fields

**Service layer (medium signal):**
- Service class names and their public method signatures
- Use case / command / query handler names (CQRS patterns)
- Business rule validators and policy classes

**Test files (excellent signal for intent):**
- Test `describe` blocks and test case names
- Integration test files that reveal end-to-end workflows
- Fixture names and factory methods that reveal domain vocabulary

### 2.3 Domain grouping

- Group by WHAT it does (business capability), not HOW it is built.
- Target **5–15 domains** for a medium-sized repo.
- Cross-cutting technical concerns (logging, error handling, config) go under `cross_cutting` — they are **not** business domains.
- Prefer fewer, broader domains over many narrow ones.

### 2.4 NFR signal scan (preliminary pass)

For each domain, record NFR signals:

| Category | Patterns to find |
|---|---|
| Performance | Cache calls, pagination params, async workers, DB index definitions, explicit timeout values |
| Security | Auth middleware/guards, role checks, password hashing (bcrypt/argon2), input validation schemas, rate limiting, CORS/CSRF config |
| Reliability | Retry logic, circuit breakers, health check endpoints, graceful shutdown handlers, dead letter queues, idempotency keys |
| Scalability | Message queue producers/consumers, stateless patterns, concurrency limits, Kubernetes HPA configs |
| Observability | Structured logging with correlation IDs, metrics emission, distributed tracing (OpenTelemetry), readiness/liveness probes |

### 2.5 Write domain-map.yaml

Create `domain-map.yaml` at the repository root:

```yaml
# domain-map.yaml
# Generated by: /opsx-full-onboard
# Status: DRAFT — review and correct domain groupings before running per-domain specs
# Generated: <today's date>

version: 1
generated: true

domains:
  - name: <lowercase-hyphenated-slug>
    label: "<Human Readable Label>"
    description: >
      <1–2 sentences. "Handles X and provides Y to Z.">
    paths:
      - "<relative/path/to/code/>"
    tech_stack:
      - "<e.g., TypeScript / Express>"
    nfr_signals:
      - type: performance|security|reliability|scalability|observability
        evidence: "<pattern found and where>"
    confidence: high|medium|low
    evidence: >
      <Brief note on what code signals led to this domain inference.>

cross_cutting:
  - name: <concern-name>
    description: "<what it does>"
    paths:
      - "<relative/path/>"

notes: >
  <Uncertainty areas, apparent dead code, unclear ownership.>
```

**After writing domain-map.yaml, stop here.** Tell the user:

> "`domain-map.yaml` has been created. **Before continuing:**
> 1. Review the domain groupings — correct any misidentified capabilities
> 2. Fill in or correct any `paths` entries that look wrong
> 3. Run `/opsx-full-onboard` again to start per-domain spec generation"

Do not proceed to Step 3 in this session.

## Step 3 — Functional Spec (one domain)

Work on the domain selected in Step 1c.

Read `domain-map.yaml` for:
- `name` (folder slug), `label`, `paths`, `description`

If `openspec/specs/<domain.name>/spec.md` exists with substantive content, confirm with the user before overwriting.

### Deep scan — functional behavior

Scan **all files** under the domain's `paths`. Focus on WHAT the system does, not HOW.

**Entry points:**
- HTTP routes: method, path, inputs (path/query/body params), outputs (response shapes), auth requirements, error responses
- Message consumers: triggering event/message, side effects, outputs
- Scheduled jobs: schedule, actions performed
- Exported library functions: inputs, outputs, guarantees
- CLI commands: flags/args, behavior

**Business rules:**
- Conditional logic (if X then Y, only allow if Z)
- Validation constraints (required fields, format rules, min/max values)
- State machine transitions (what states exist? what triggers transitions?)
- Authorization rules (who can do what, under what conditions?)
- Calculation or transformation rules

**Data contracts:**
- What data shapes flow in and out? (describe data, not class names)
- What external systems are called? (describe the interaction, not the library)
- What is persisted and what are the persistence semantics?

**Error conditions:**
- Explicitly handled error cases (cause and response)
- Validation failure messages
- Retry or fallback behavior on external failures

**Test files:** Read test files in or near the domain paths. Use test descriptions to verify inferences and find edge cases you might have missed.

### Write `openspec/specs/<domain.name>/spec.md`

```markdown
# <Domain Label> Specification

## Purpose
<2–3 paragraphs. Business perspective: who uses it, what problem it solves, what value it delivers.
Do NOT describe classes, databases, or implementation technology.>

## Scope

**In scope:**
- <Bullet list of what IS covered by this spec>

**Out of scope:**
- <Bullet list of what is explicitly NOT covered>

## Requirements

### Requirement: <Behavior Name>
The system SHALL <observable behavior statement>.

> Rationale: <optional — why this requirement exists, if non-obvious>

#### Scenario: <Happy path scenario name>
- GIVEN <precondition>
- WHEN <trigger>
- THEN <primary outcome>
- AND <secondary outcome, if applicable>

#### Scenario: <Error or edge case name>
- GIVEN <precondition>
- WHEN <trigger>
- THEN <outcome>
```

**Rules:**
- Use **SHALL / MUST** for absolutes. **SHALL NOT / MUST NOT** for prohibitions. **SHOULD** for strong recommendations. **MAY** for optional.
- Requirements must describe **externally observable behavior** only.
- Scenarios must be independently testable.
- No class names, function names, library names, or table names in spec.md.
- Mark uncertainty: `<!-- TODO(EM-REVIEW): inferred — verify this behavior -->`

## Step 4 — NFR Spec (same domain, same session)

Immediately after writing the functional spec, scan the same domain paths for NFR signals and append a `## Non-Functional Requirements` section.

Use `nfr_signals` from `domain-map.yaml` as a starting point, then scan deeper.

Only generate NFR sections for categories where code signals were actually found. Do not add empty boilerplate.

**Append to the same `openspec/specs/<domain.name>/spec.md`:**

```markdown
---

## Non-Functional Requirements

### Performance

#### Requirement: <Performance Behavior Name>
The system SHALL <observable performance behavior>.

> **Evidence**: <file/module>
> **Target**: **[PLACEHOLDER — EM: replace with actual target, e.g., "p95 < 200ms"]**

##### Scenario: Normal operating load
- GIVEN the service is handling normal traffic
- WHEN <specific action> is performed
- THEN the response is delivered within [PLACEHOLDER] at the 95th percentile

---

### Security

#### Requirement: <Security Behavior Name>
The system SHALL <observable security behavior>.

> **Evidence**: <file/module>

##### Scenario: Unauthenticated access attempt
- GIVEN a request with no valid credentials
- WHEN <action> is attempted
- THEN the system SHALL reject the request with [PLACEHOLDER — HTTP 401 or equivalent]

---

### Reliability

#### Requirement: <Reliability Behavior Name>
The system SHALL <observable reliability behavior>.

> **Evidence**: <file/module>
> **Target**: **[PLACEHOLDER — EM: e.g., "99.9% monthly uptime"]**

---

### Observability

#### Requirement: <Observability Behavior Name>
The system SHALL <observable observability behavior>.

> **Evidence**: <file/module>
```
