Perform a **capability discovery analysis** of this codebase. Your goal is to identify what business capabilities the software provides and organise them into logical domains suitable for OpenSpec specification.

**This is a read-then-write task.** Read and analyse the codebase extensively before creating any files.

---

## Phase 1 — Repository Structure Scan

Explore the repository layout:

1. List all top-level directories and identify their roles (service? library? app? infrastructure? tooling?)
2. For each significant folder, find project manifests: `package.json`, `*.csproj`, `*.sln`, `pom.xml`, `build.gradle`, `Pipfile`, `pyproject.toml`, `go.mod`, `Cargo.toml`, `pubspec.yaml`, `Gemfile`
3. Look for existing documentation: `README.md`, `docs/`, `ARCHITECTURE.md`, `ADR/`, `openapi.yaml`, `swagger.json`
4. Note which technology stacks are present across different parts of the repo

---

## Phase 2 — Feature and Capability Extraction

For each significant project or module, extract capabilities by examining:

**API surfaces (highest signal):**
- HTTP routes and controllers — the URL path reveals the domain (e.g., `/api/payments/` → payments)
- GraphQL schemas and resolver names
- gRPC `.proto` service and method definitions
- Message queue topic/channel names and their consumer handlers
- CLI command definitions and their descriptions
- Scheduled job names and handler function names

**Domain models (medium signal):**
- Entity class names, aggregate roots, value objects
- Database table/collection names (migration files, ORM models, schemas)
- DTO, request, and response type names and their fields

**Service layer (medium signal):**
- Service class names and their public method signatures
- Use case / command / query handler names (CQRS patterns)
- Business rule validators and policy classes

**Test files (excellent signal for intent):**
- Test `describe` blocks and test case names often read as plain-English requirements
- Integration test files reveal end-to-end workflows
- Fixture names and factory methods reveal domain vocabulary

---

## Phase 3 — Domain Grouping

Group extracted capabilities into **business capability domains**:

- Think like a business stakeholder: what capabilities does this system provide to its users?
- Group by WHAT it does, not by HOW it is built (e.g., `UserController` + `UserService` + `users` DB table → one "user-management" domain)
- Target **5–15 domains** for a medium mono-repo. Too many makes spec maintenance hard.
- Cross-cutting technical concerns (logging, error handling, config, shared utilities) are **not** business domains — list them separately under `cross_cutting`
- When uncertain, prefer fewer broader domains over many narrow ones

---

## Phase 4 — NFR Signal Detection

For each domain, scan for non-functional requirement signals:

| Category | Look for |
|---|---|
| **Performance** | Cache decorators or calls (Redis, Memcached, `@Cacheable`), pagination patterns, async workers, DB index definitions, connection pool config, explicit timeout values |
| **Security** | Auth middleware or guards (`[Authorize]`, `@UseGuards`, JWT validation), role/permission checks, password hashing (bcrypt/argon2), input validation schemas (Joi, Zod, FluentValidation), rate limiting middleware, CORS/CSRF config |
| **Reliability** | Retry logic with backoff, circuit breakers (`Polly`, `@CircuitBreaker`), health check endpoints (`/health`, `/ready`), graceful shutdown handlers, dead letter queues, idempotency keys |
| **Scalability** | Message queue producers/consumers (Kafka, RabbitMQ, SQS), stateless design patterns, concurrency limits, worker pool sizes, autoscaling config |
| **Observability** | Structured logging with correlation IDs, metrics emission (Prometheus, StatsD), distributed tracing (OpenTelemetry), health/readiness probes |

## Phase 5 — Create Output Artifacts

### Artifact 1: Create `domain-map.yaml` at the repository root

```yaml
# domain-map.yaml
# Generated by: opsx-discover-capabilities
# Status: DRAFT — review and edit before running setup
# Generated: <today's date>

version: 1
generated: true

domains:
  - name: <lowercase-hyphenated-slug>            # e.g., auth, payments, notifications
    label: "<Human Readable Label>"              # e.g., "Authentication & Session Management"
    description: >
      <1–2 sentences describing what this domain does from a business perspective.
      Write as: "Handles X and provides Y to Z.">
    paths:
      - "<relative/path/to/code/>"              # All code paths that belong to this domain
    tech_stack:
      - "<e.g., TypeScript / Express>"
    nfr_signals:
      - type: performance|security|reliability|scalability|observability
        evidence: "<brief note: what pattern was found and in which file/module>"
    confidence: high|medium|low
    evidence: >
      <Brief note on what code signals (route paths, class names, table names, test names, etc.)
      led to this domain inference. Be specific.>

cross_cutting:
  - name: <concern-name>                         # e.g., logging, error-handling, config
    description: "<what it does>"
    paths:
      - "<relative/path/>"

notes: >
  <Any observations that don't fit into domains: areas of uncertainty, apparent dead code,
  components whose purpose is unclear, questions for the EM.>
```

## Guidelines

- **Do not invent capabilities.** Only infer from code evidence. Mark unclear components `confidence: low`.
- **Be thorough in Phase 2.** Shallow scanning produces shallow domain maps. Read actual handler/controller code, not just file names.
- **One domain = one future spec file.** Don't create a domain you wouldn't want to maintain a spec for.
- **Uncertainty is useful.** Add `confidence: low` entries with honest `evidence` notes — the EM will clarify them.
- **Test files are goldmines.** If a component has good test coverage, test names often express the requirements more clearly than the implementation code does.
